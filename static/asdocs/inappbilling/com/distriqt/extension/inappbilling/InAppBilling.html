consumePurchase()method 
public function consumePurchase(purchase:Purchase):Boolean

Call this function to consume a purchase, making it available for purchase in the store again.

This is not required for Apple's InApp Purchases however calling this function will not affect your application, so you are suggested to use this mechanism to provide cross-platform compatibility. Under iOS this call will immediately fire the consume success event.

Security Recommendation: You must send a consumption request before provisioning the benefit of the consumable purchase to the user. Make sure that you have received a successful consumption response before you provision the item.

Events

Calling this function may potentially dispatch any of the following:

  • InAppBillingEvent.CONSUME_SUCCESS
  • InAppBillingEvent.CONSUME_FAILED

Parameters

purchase:Purchase — A Purchase object with a valid productId

Returns
Boolean — true if the consumption process was started successfully and false if there was an issue with the supplied parameters or the service is not setup

See also

These must be setup in the store previously through and you must have at least one product defined:

makePurchase()method 
public function makePurchase(request:PurchaseRequest):Boolean

Make a purchase of a specified product. The purchase must contain a valid product id that has been previously retrieved by use of the #getProducts function, and a valid quantity (greater than zero).

If this function returns false during development, it's likely due to the fact that your application hasn't been setup correctly. It will either be that the application descriptor hasn't been correctly setup, or the application hasn't been correctly setup in the appropriate application store. Double check all of the application setup and try again.

If it returns false once deployed, it's likely that the connection to the application store has failed. You could com.distriqt.test the devices network connection, inform the user of the failure or just wait and try again at a later point.

Events

  • PurchaseEvent.

Parameters

request:PurchaseRequest — The PurchaseRequest to make. It must have a valid product id and quantity specified.

Returns
Boolean — true if the purchase was started successfully and false otherwise

restorePurchases()method 
public function restorePurchases(applicationUsername:String):void

This function will restore any previously made purchases by this user. This is done by contacting the server and for each purchase made the appropriate event will fire again, ie. for a successful purchase PurchaseEvent.PURCHASE_SUCCESS will fire.

Once all the purchase events have fired you will receive an InAppBillingEvent.RESTORE_PURCHASES_SUCCESS to indicate the process has completed

Users restore transactions to maintain access to content they've already purchased. For example, when they upgrade to a new phone, they don’t lose all of the items they purchased on the old phone. Include some mechanism in your app to let the user restore their purchases, such as a Restore Purchases button. Restoring purchases prompts for the user’s App Store credentials, which interrupts the flow of your app: because of this, don’t automatically restore purchases, especially not every time your app is launched.

Events

Calling this function may potentially dispatch any of the following:

  • InAppBillingEvent.RESTORE_PURCHASES_SUCCESS
  • InAppBillingEvent.RESTORE_PURCHASES_FAILED

Parameters

applicationUsername:String — Should be the same username that you used in your PurchaseRequest

setup()method 
public function setup(keyOrBillingService:*):Boolean

This function sets up the connection to the application store, setting up any required security connections.

You should listen and wait for the SETUP_SUCCESS and SETUP_FAILURE events before proceeding to any further extension calls.

iOS InApp Purchases:

This will register the purchase observer. This will make any pending purchase events to be dispatched, so you can handle any interrupted purchases. To do this you should add your PURCHASE_event listeners before calling this function.

Google Play InApp Billing:

The Google Play Public Key parameter on the Billing Service should be your Google Play License Key for the application. See the main documentation for more information on acquiring this key from the Google Play Developer Console. Please note it's highly recommended that you attempt to obfuscate this key in your code. See the security recommendations in the main documentation.

Events

Calling this function may potentially dispatch any of the following:

  • PurchaseEvent.
  • InAppBillingEvent.SETUP_SUCCESS
  • InAppBillingEvent.SETUP_FAILURE

purchases:updated Event  
Event Object Type: com.distriqt.extension.inappbilling.events.PurchaseEvent
PurchaseEvent.type property = com.distriqt.extension.inappbilling.events.PurchaseEvent.PURCHASES_UPDATED

Dispatched whenever a purchase state has changed. This can be when the purchases queue is updated, when a purchase transaction is complete or when a user cancels a purchase.

You should use this event to iterate over the Purchase objects contained in the data array using the Purchase.transactionState to process the purchase state. 

         private function purchasesUpdatedHandler( event:PurchaseEvent ):void
         {
          for each (var purchase:Purchase in event.data)
          {
              switch (purchase.transactionState)
              {
                  case case Purchase.STATE_FAILED:
                      break;
                           case Purchase.STATE_PURCHASED:
                      break;
                           // ETC...
              }
          }
         }
should:add:purchase Event  
Event Object Type: com.distriqt.extension.inappbilling.events.PurchaseRequestEvent
PurchaseRequestEvent.type property = com.distriqt.extension.inappbilling.events.PurchaseRequestEvent.SHOULD_ADD_PURCHASE

Dispatched when a purchase request is going to be added to the queue via an external source, such as through the AppStore promotions functionality. The request will contain information about the product id that is being requested.

You should respond to this event within a short amount of time to indicate if the purchase should be added to the purchase queue and the purchase flow initiated. This is done by calling InAppBilling.service.shouldAppPurchase

If you don't respond to this event the default response is to add the purchase to the queue and start the purchase flow.
